# 产品介绍文章编写规则

## 文章定位

产品介绍文章旨在让读者理解一个项目/产品的价值、设计思路和使用方法。重点在于**讲好故事**、**展示价值**，以及**降低使用门槛**。

## 核心原则

### 1. 故事化引入，而非功能堆砌
- ✅ 从痛点或背景故事开始
- ✅ 说明"为什么需要这个产品"而非"这个产品有什么功能"
- ✅ 用用户视角描述问题和解决方案
- ❌ 避免上来就罗列功能清单
- ❌ 避免过度技术化的描述

**示例对比：**

❌ **功能堆砌式（避免）：**
```
Cogita 是一个博客框架，具有以下功能：
1. 支持 Markdown 写作
2. 自动生成路由
3. 主题系统
4. 插件扩展
...
```

✅ **故事化引入（推荐）：**
```
## 背景：为什么需要 Cogita？

当我尝试用 Rspress 搭建个人博客时，发现虽然它功能强大，
但要配置一个完整的博客仍然需要手动完成大量工作：主题开发、
内容管理、配置整合...这个过程不仅繁琐，而且对于只想专注于
内容创作的开发者来说，门槛相对较高。

Cogita 的诞生就是为了解决这个问题...
```

### 2. 分层次展示信息
- ✅ 背景和愿景 → 核心架构 → 使用方法 → 未来规划
- ✅ 每个层次针对不同读者群体
- ✅ 让想快速上手的人能跳到"使用方法"章节
- ✅ 让想深入了解的人能看懂"架构设计"
- ❌ 避免所有信息混在一起
- ❌ 避免缺少快速开始指南

**推荐结构：**
```
## 背景：为什么需要它？（吸引兴趣）
讲述痛点和产品诞生的故事...

## 架构设计：它的内在机制（建立信任）
解释核心设计理念和技术方案...

## 实际使用：三步开启（降低门槛）
提供最简单的上手步骤...

## 未来展望：蛮荒阶段与无限可能（激发期待）
说明路线图和发展方向...
```

### 3. 价值主张清晰
- ✅ 明确说明产品的核心价值（节省时间、降低成本、提升体验等）
- ✅ 用对比来突出优势（vs 其他方案、vs 手动实现）
- ✅ 用具体数据或场景说明价值（如：从配置 2 天到 5 分钟）
- ❌ 避免空洞的形容词（"强大"、"优秀"、"完美"）
- ❌ 避免过度营销化的语言

**价值表达示例：**
```
❌ "Cogita 是一个强大且易用的博客框架"
✅ "Cogita 的核心目标是：成为一个基于 Rspress、无需繁琐配置、
真正开箱即用的静态博客框架。你只需要安装它，选择一个喜欢的主题，
然后就可以立即开始写作。"
```

### 4. 快速开始的可操作性
- ✅ 提供完整的安装和配置代码
- ✅ 用"三步"、"五步"这样的明确步骤
- ✅ 代码可以直接复制粘贴运行
- ✅ 说明预期效果（"运行后会看到..."）
- ❌ 避免步骤不完整或跳跃
- ❌ 避免假设读者已经配置了某些环境

**快速开始示例：**
```
## 实际使用：三步开启你的博客

1. 安装依赖：`pnpm add @cogita/core @cogita/theme-lucid`

2. 创建配置文件 `cogita.config.ts`：
   ```typescript
   // 完整可运行的代码
   ```

3. 在 `posts` 目录下开始撰写你的 Markdown 文章。

就是这样！运行 `cogita dev`，一个漂亮的博客网站就已经在本地运行起来了。
```

### 5. 架构设计的解释
- ✅ 用类比或图示说明设计理念
- ✅ 说明每个核心组件的作用和价值
- ✅ 突出设计的独特性和优势
- ✅ 可以用列表罗列组件，但要配合段落解释
- ❌ 避免只有技术术语没有解释
- ❌ 避免过度技术细节（放到技术文档里）

**架构描述示例：**
```
## 架构设计：Cogita 的内在机制

Cogita 的设计哲学是"约定优于配置"，它通过一套精心设计的架构，
将博客系统的复杂性封装起来，为用户提供最简洁的接口。

其核心由以下几个部分组成：

1. **`@cogita/core`**: 这是框架的大脑。它作为 Rspress 的一层封装，
   负责加载用户的配置文件，并智能地协调主题与插件的工作...
   
2. **主题系统**: 不仅仅是 UI 皮肤。每个主题都是一个独立的 NPM 包...
```

### 6. 未来规划和社区
- ✅ 坦诚说明当前阶段（如"蛮荒阶段"、"MVP 版本"）
- ✅ 列出未来规划，激发期待
- ✅ 邀请参与和贡献
- ✅ 保持谦逊和开放的态度
- ❌ 避免过度承诺（"未来会支持一切"）
- ❌ 避免对当前问题避而不谈

**未来展望示例：**
```
## 未来展望：蛮荒阶段与无限可能

Cogita 目前还处于非常早期的"蛮荒阶段"，许多功能仍在探索和开发之中。
但这同样也意味着它拥有无限的可能性。在未来的路线图中，我们计划实现：

- 更丰富的主题生态
- 更强大的插件系统
- 国际化 (i18n)

如果你对这个项目感兴趣，欢迎关注我们的进展，甚至参与到贡献中来！
```

## 文章结构模板

### 开头（引入问题）
- 描述用户的痛点或需求
- 说明现有方案的不足
- 引出产品的诞生背景

### 核心价值（解决方案）
- 用一句话概括产品的核心价值
- 说明它如何解决前面提到的问题
- 突出与其他方案的差异

### 架构设计（建立信任）
- 解释设计理念（如"约定优于配置"）
- 介绍核心组件和它们的作用
- 说明技术方案的优势

### 快速开始（降低门槛）
- 提供最简单的安装和配置步骤
- 确保代码可以直接运行说明预期效果

### 未来规划（激发期待）
- 坦诚说明当前状态
- 列出未来计划
- 邀请参与和反馈

## Frontmatter 格式

```yaml
---
title: "产品名：核心价值一句话概括"
createDate: "YYYY-MM-DD"
updateDate: "YYYY-MM-DD"
tags:
  - 产品名或项目名
  - 技术栈标签
  - 领域标签（如 博客框架、开发工具）
  - 不超过 5 个
author: "作者名"
excerpt: "用一段话说明产品是什么、解决什么问题、核心特点。类似于电梯演讲。"
---
```

## 语言风格

- **热情但不夸张**：展现对产品的热爱，但避免过度营销
- **用户视角**：多用"你"、"你的"，站在用户角度描述
- **简洁直接**：避免复杂的长句和技术术语堆砌
- **谦逊开放**：承认不足，欢迎反馈和贡献

## 常用表达模式

### 引入问题
```
"当我尝试用 X 做 Y 时，我发现..."
"虽然 X 很强大，但要实现 Y，仍然需要..."
"在使用 X 的过程中，开发者经常遇到..."
```

### 说明价值
```
"它的核心目标是..."
"通过 X，你可以..."
"这意味着..."
```

### 邀请参与
```
"如果你对这个项目感兴趣，欢迎..."
"我们期待..."
"欢迎关注我们的进展，甚至参与到贡献中来！"
```

## 检查清单

- [ ] 是否从用户痛点而非功能清单开始？
- [ ] 核心价值是否清晰且有说服力？
- [ ] 快速开始的代码是否完整可运行？
- [ ] 架构设计是否既有深度又易懂？
- [ ] 是否坦诚说明了当前的不足？
- [ ] 未来规划是否具体可信？
- [ ] 语气是否热情但不过度营销？
- [ ] 读者看完后是否想尝试使用？
